I've restructured React codebases more times than I'd like to admit. Somewhere around the third or fourth time you find yourself grepping for where a component actually lives, you start thinking hard about architecture. Over the past few years I've worked with two approaches that keep coming up in conversations: Feature-Sliced Design (FSD) and what I'll loosely call modular architecture. I've shipped production code with both, and I have opinions.

## What Feature-Sliced Design actually is

FSD came out of the Russian-speaking frontend community and it's more prescriptive than most people expect. The core idea is that your codebase has strict layers — `app`, `pages`, `widgets`, `features`, `entities`, `shared` — and each layer can only import from layers below it. Within each layer, you organize by business slice.

The folder structure looks something like this:

```plaintext
/src
  /app
  /pages
    /dashboard
  /widgets
    /user-menu
  /features
    /edit-profile
      /ui
        EditProfileForm.tsx
      /model
        useEditProfile.ts
      /api
        updateProfile.ts
  /entities
    /user
      /ui
        UserAvatar.tsx
      /model
        types.ts
      /api
        getUser.ts
  /shared
    /ui
    /lib
    /api
```

The strict layering is the whole point. A feature can use entities and shared code, but it can never import from another feature. A widget can compose features, but features don't know about widgets. This is enforced — ideally with lint rules, not just good intentions.

I personally find the layer hierarchy elegant on paper. Where it gets tricky is the `features` vs `entities` distinction. The FSD docs define entities as "business objects" and features as "user actions on those entities." So `user` is an entity, but `editProfile` is a feature. Makes sense until you're staring at some piece of code that fetches a user's subscription status and renders a badge — is that an entity concern or a feature? You'll spend time on these taxonomy debates, especially with a team that's new to FSD.

Here's roughly what a feature looks like in practice:

```typescript
// features/edit-profile/model/useEditProfile.ts
import { useMutation, useQueryClient } from '@tanstack/react-query';
import { updateProfile } from '../api/updateProfile';
import type { User } from '@/entities/user';

export function useEditProfile() {
  const queryClient = useQueryClient();

  return useMutation({
    mutationFn: (data: Partial<User>) => updateProfile(data),
    onSuccess: () => {
      queryClient.invalidateQueries({ queryKey: ['user', 'me'] });
    },
  });
}
```

```tsx
// features/edit-profile/ui/EditProfileForm.tsx
import { useEditProfile } from '../model/useEditProfile';

export function EditProfileForm({ user }: { user: User }) {
  const { mutate, isPending } = useEditProfile();

  const handleSubmit = (e: React.FormEvent<HTMLFormElement>) => {
    e.preventDefault();
    const formData = new FormData(e.currentTarget);
    mutate({ displayName: formData.get('displayName') as string });
  };

  return (
    <form onSubmit={handleSubmit}>
      <input name="displayName" defaultValue={user.displayName} />
      <button disabled={isPending}>
        {isPending ? 'Saving...' : 'Save'}
      </button>
    </form>
  );
}
```

Nothing revolutionary in the code itself. The value is structural — you know exactly where this lives and what it's allowed to touch.

## Where FSD breaks down

The layer rules are strict, and that strictness cuts both ways. The one that bit me hardest: cross-feature communication. Say you have a `notifications` feature and an `edit-profile` feature, and after a successful profile edit you want to show a toast notification. In FSD, features can't import from each other. So you end up threading this through a shared event bus, or lifting the orchestration up to the widget/page layer.

That's fine architecturally, but in practice it means your page components accumulate a lot of wiring logic. You end up with pages that are basically just glue code connecting features together, and that glue gets messy fast when you have complex user flows.

The other friction point is `shared`. Everything that doesn't fit neatly into the hierarchy gets dumped into `shared`, and without discipline it becomes a junk drawer. I've seen `shared/lib` folders with 40+ files that have nothing in common except "didn't belong anywhere else."

## Modular architecture — the pragmatic alternative

What I'm calling modular architecture is less formal. There's no canonical spec like FSD has. The basic idea: group code by domain module, each module owns its own components, hooks, utils, types, and API calls. Modules can have explicit public APIs (barrel exports) and you enforce boundaries between them.

```plaintext
/src
  /modules
    /auth
      /components
        LoginForm.tsx
        AuthGuard.tsx
      /hooks
        useAuth.ts
        useSession.ts
      /api
        auth.ts
      /types.ts
      index.ts        ← public API
    /billing
      /components
      /hooks
      /api
      index.ts
    /dashboard
      /components
      /hooks
      index.ts
  /shared
    /components
    /hooks
    /utils
```

The key difference from FSD: there's no fixed layer hierarchy. You decide on dependency rules per-project. Maybe `auth` is a foundational module that others can import from. Maybe `billing` and `dashboard` are leaf modules that shouldn't know about each other. You document this and enforce it with tooling.

I find this works better for teams that are already mid-project and can't afford to reclassify their entire codebase into entities/features/widgets. You can adopt it incrementally — pull one messy domain out into a clean module, set up its public API, add a lint rule that blocks deep imports, and move on.

A module's public API is just a barrel file, but it matters a lot:

```typescript
// modules/auth/index.ts
export { LoginForm } from './components/LoginForm';
export { AuthGuard } from './components/AuthGuard';
export { useAuth } from './hooks/useAuth';
export { useSession } from './hooks/useSession';
export type { User, Session } from './types';
```

Then in your eslint config (using `eslint-plugin-boundaries` or `eslint-plugin-import`), you restrict imports so other modules can only use what's exported from `index.ts`. No reaching into `modules/auth/hooks/useAuth` directly — you go through the barrel.

```javascript
// eslint config (flat config style)
import boundaries from 'eslint-plugin-boundaries';

export default [
  {
    plugins: { boundaries },
    rules: {
      'boundaries/element-types': ['error', {
        default: 'disallow',
        rules: [
          { from: 'module', allow: ['shared'] },
          { from: 'module', allow: ['module'], importKind: 'value',
            condition: { targetNotSelf: true } },
        ],
      }],
    },
  },
];
```

The config gets more nuanced than this in real life, but you get the idea. The point is that boundaries are enforced by tooling, not by hope.

## My honest comparison

I prefer modular architecture for most teams and most projects. Here's why.

FSD is great if you're starting fresh with a team that buys into it fully. The rigid structure means less bikeshedding about where things go — the spec tells you. New developers can onboard by reading the FSD docs and immediately understand the project layout. That's genuinely valuable.

But FSD's rigidity becomes a liability when your domain doesn't map cleanly to its layer model. I worked on a project where we had complex multi-step workflows that spanned several "features" and "entities," and the layer rules made the code harder to follow, not easier. We were constantly fighting the architecture instead of using it.

Modular architecture gives you the benefits of isolation and clear boundaries without prescribing how deep the hierarchy goes. You get to define the rules that make sense for your project. The downside is that you need a team that has the discipline to actually define and maintain those rules. Without that, modules degrade into the same mess you started with — just with extra folders.

Here's roughly how I think about it:

**Pick FSD if:** you're starting a new project, the team is willing to learn the spec, and your domain maps well to the entity/feature split. Also if you want to hire or scale the team quickly — the standardization helps.

**Pick modular if:** you're refactoring an existing codebase, your domain is complex or doesn't fit neat categories, or your team prefers defining their own conventions. Also if you're working across multiple projects — modular patterns transfer more easily because they're not tied to one spec.

## The migration problem

Regardless of which you pick, migrating an existing codebase is the hard part. Not technically hard — it's mostly moving files around and updating imports. The hard part is doing it without blocking feature work for weeks.

What worked for me: don't try to migrate everything at once. Pick the most painful module — the one where developers keep breaking things because of tangled dependencies — and extract that first. Get the lint rules in place for that one module. Ship features on top of the new structure so the team sees the benefits. Then do the next one.

Circular dependencies will surface during migration and they're annoying to untangle. Turn on `import/no-cycle` early, even before you start restructuring:

```javascript
rules: {
  'import/no-cycle': ['error', { maxDepth: 3 }],
}
```

I set `maxDepth` because without it the rule can get slow on large codebases — it has to trace every possible import chain. Depth 3 catches the vast majority of real circular deps without tanking your lint times.

One thing that surprised me: the biggest resistance to architectural changes wasn't technical, it was organizational. Developers had mental models of "this file is over here" built up over months, and moving things around broke that. Clear communication and a migration guide that maps old paths to new paths made a real difference.

## Lazy loading at module boundaries

Whatever architecture you choose, module boundaries are natural code-splitting points. I lazy load at the route level and sometimes at the module level for heavy features:

```tsx
const BillingPage = lazy(() => import('@/modules/billing/pages/BillingPage'));
const DashboardPage = lazy(() => import('@/modules/dashboard/pages/Dashboard'));

function AppRoutes() {
  return (
    <Suspense fallback={<PageSkeleton />}>
      <Routes>
        <Route path="/billing" element={<BillingPage />} />
        <Route path="/dashboard" element={<DashboardPage />} />
      </Routes>
    </Suspense>
  );
}
```

Don't over-split. I made this mistake once — lazy loaded individual components within a page and ended up with a waterfall of network requests that made the page feel slower, not faster. Route-level splitting is almost always enough. If you need more granular splitting, do it for genuinely heavy components (rich text editors, chart libraries) and nothing else.

## What I'd do on my next project

If I started a new large React app tomorrow, I'd probably go with a modular approach but borrow some ideas from FSD. Specifically, I'd keep FSD's concept of a `shared` layer and the rule that shared code can't import from modules. I'd set up `eslint-plugin-boundaries` on day one, before writing any feature code. And I'd define module boundaries based on team ownership — if two developers own billing, all billing code lives in one module.

I wouldn't adopt full FSD unless the team had used it before, because the learning curve is real and the taxonomy debates slow you down in the early weeks when you should be shipping.

The architecture that works best is the one your team will actually maintain. Pick one, enforce it with tooling, and revisit it when it starts hurting. That's about as honest as advice gets on this topic.
