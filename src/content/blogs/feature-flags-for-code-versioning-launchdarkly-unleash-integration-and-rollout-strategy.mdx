I've shipped features behind flags on three different projects now, across two different platforms (LaunchDarkly and Unleash), and my feelings about feature flags are… complicated. They're genuinely useful, but they also introduce a kind of complexity that sneaks up on you if you're not careful. Here's what I've learned from actually using them in production.

## LaunchDarkly vs Unleash — which one and why

I've used both. On a previous project we ran Unleash because the team wanted to self-host everything. On a more recent e-commerce project we went with LaunchDarkly. They solve the same core problem but the experience is pretty different.

**LaunchDarkly** is polished. The dashboard is good, the SDK docs are solid, targeting rules are powerful out of the box, and you get things like experimentation and audit logs without bolting on extra tooling. The downside is cost — it gets expensive fast, especially once you have a decent number of monthly active users. We hit a billing surprise about four months in when our MAU count crossed a tier boundary. Nobody had been watching that number.

**Unleash** is more bare-bones but you own the infrastructure. The open-source version covers basic toggles and gradual rollouts well enough. Where it falls short is advanced targeting — if you need to target by user attributes, subscription tier, geo, etc., you either write custom activation strategies or pay for Unleash Enterprise. The admin UI is functional but not great. I also ran into a few issues with the React SDK where flag updates didn't propagate as quickly as I expected on the client side.

I personally prefer LaunchDarkly for teams that can afford it, mainly because the targeting and the React SDK integration are smoother. But if cost is a concern or your use case is mostly simple on/off toggles, Unleash with self-hosting is perfectly fine.

## Setting up flags in React

The LaunchDarkly React SDK gives you a `useFlags` hook and a `useLDClient` hook. One thing I see people get wrong (and I got wrong the first time) is trying to destructure `ldClient` out of `useFlags`. That's not how it works — `useFlags` just returns the flag values as an object.

```typescript
import { useFlags, useLDClient } from 'launchdarkly-react-client-sdk';

function useFeatureFlag(flagKey: string, defaultValue: boolean = false) {
  const flags = useFlags();
  return flags[flagKey] ?? defaultValue;
}

// if you need the client instance for tracking or identifying users:
function useFeatureFlagWithTracking(flagKey: string) {
  const flags = useFlags();
  const ldClient = useLDClient();

  const trackConversion = (metricKey: string) => {
    ldClient?.track(metricKey, { flagKey });
  };

  return {
    enabled: flags[flagKey] ?? false,
    trackConversion,
  };
}
```

I wrap flags in custom hooks like this not because it's architecturally clever, but because flag keys are strings and I don't want them scattered across fifty components. When you inevitably need to rename or retire a flag, having one place to grep is worth the minor abstraction.

For TypeScript, I define the flag shape upfront:

```typescript
interface AppFeatureFlags {
  'checkout-v2': boolean;
  'new-search-algo': boolean;
  'promo-banner-holiday': boolean;
}
```

This doesn't give you runtime safety — the SDK still returns `LDFlagSet` which is basically `Record<string, any>` — but it helps catch typos during development and gives you autocomplete if you cast the return value.

## Conditional rendering without making a mess

The obvious pattern is ternaries in JSX:

```tsx
function ProductPage() {
  const showNewCheckout = useFeatureFlag('checkout-v2');

  return (
    <div>
      <ProductDetails />
      {showNewCheckout ? <CheckoutV2 /> : <LegacyCheckout />}
    </div>
  );
}
```

This is fine for one or two flags. It becomes a problem when you have four flags active on the same page and your render function turns into a decision tree. On one project we had a page with three overlapping flags and the component was basically unreadable. What worked better was pulling the flag checks up and passing resolved props down:

```tsx
function ProductPageContainer() {
  const showNewCheckout = useFeatureFlag('checkout-v2');
  const showNewSearch = useFeatureFlag('new-search-algo');

  return (
    <ProductPage
      checkoutComponent={showNewCheckout ? CheckoutV2 : LegacyCheckout}
      searchComponent={showNewSearch ? SearchV2 : LegacySearch}
    />
  );
}
```

Not revolutionary, but it keeps the actual page component clean and testable. You can test `ProductPage` without mocking any flag provider.

## Rollout strategy — the part that actually matters

The code integration is honestly the easy part. The rollout strategy is where things get interesting and where I've seen teams mess up.

### Percentage rollouts

LaunchDarkly handles percentage rollouts natively — you don't need to write your own random number logic. You just configure "serve `true` to 5% of users" in the dashboard and it handles the hashing so the same user consistently gets the same variation. This is important: you want sticky assignment, not random-on-every-request assignment. If a user refreshes and suddenly the feature disappears, you'll get confused bug reports.

We typically started at 5%, watched error rates and performance metrics for 48 hours, then bumped to 25%, then 50%, then 100%. The 48-hour window caught issues that quick smoke tests missed — things like edge cases in specific browsers, interactions with cached data, or problems that only showed up under certain user states.

### Targeting specific segments first

Before doing a percentage rollout to random users, we'd target internal users and beta testers first. LaunchDarkly lets you create user segments and target them with rules. We had a segment for employees (matched on email domain) and another for users who had opted into beta features (a custom attribute we sent during identification).

One thing that surprised me: you need to be careful about the order of targeting rules. Rules are evaluated top-to-bottom and the first match wins. I once had a "disable for users in Germany" rule below the percentage rollout rule, which meant German users were still getting the feature through the percentage allocation. The fix was just reordering the rules, but it took us a while to figure out why our GDPR-related exclusion wasn't working.

## The flicker problem

This one is annoying and not talked about enough. When you evaluate flags client-side, there's a moment between the initial render and when the SDK finishes initialization where you're rendering with default values. If the default is `false` and the flag resolves to `true`, the user sees the old UI flash before the new one appears.

The LaunchDarkly React SDK supports bootstrapping — you can pass initial flag values as props to the provider:

```tsx
import { LDProvider } from 'launchdarkly-react-client-sdk';

// fetch flags server-side during SSR or in a loader
const serverFlags = await getServerSideFlags(userId);

<LDProvider
  clientSideID="your-client-id"
  context={{ kind: 'user', key: userId }}
  options={{ bootstrap: serverFlags }}
>
  <App />
</LDProvider>
```

The `bootstrap` option makes the SDK start with those values immediately, so there's no flash. The SDK still connects to LaunchDarkly in the background and will update if the flag value has changed since the server-side fetch, but the initial render is correct.

If you're not doing SSR, another approach is to delay rendering the flagged component until the SDK is ready. The `useLDClient` hook gives you the client instance and you can check `ldClient.waitForInitialization()`. It's not ideal — you're trading flicker for a loading state — but sometimes that's the better UX.

## Flag lifecycle — a.k.a. the tech debt nobody budgets for

Here's the uncomfortable truth about feature flags: adding them is easy, removing them is a chore that nobody wants to do. On one project we accumulated 47 flags over about 8 months. At least 30 of them were fully rolled out and serving `true` to 100% of users. They were dead code wrapped in conditionals that nobody removed because "it works, why touch it."

The problem is that dead flags make the codebase harder to reason about. New team members don't know if a flag is still being experimented with or if it's been permanent for six months. And the old code path behind the `false` branch? It's rotting. Nobody's testing it anymore, but it's still there, adding to bundle size and cognitive load.

What helped us was a simple rule: when a flag reaches 100% and stays there for two weeks with no issues, a ticket gets auto-created to remove it. We added a `createdAt` field to our flag type definition and wrote a lint rule that warned if a flag was older than 60 days without being marked as permanent.

```typescript
interface FlagMetadata {
  key: string;
  createdAt: string;
  owner: string;
  type: 'temporary' | 'permanent';
  jiraTicket: string;
}

const FLAG_REGISTRY: FlagMetadata[] = [
  {
    key: 'checkout-v2',
    createdAt: '2024-11-01',
    owner: 'akin',
    type: 'temporary',
    jiraTicket: 'FE-1234',
  },
];
```

The `jiraTicket` for removal gets created at the same time as the flag itself. This was the single most effective thing we did to manage flag debt. If the removal ticket doesn't exist, the flag doesn't get merged.

## Things that bit us

**Stale flag cache on mobile web.** We had users on mobile Safari who were getting served a stale flag value because the service worker was caching the LaunchDarkly streaming connection response. Took us a week to figure out. The fix was excluding the LaunchDarkly endpoints from the service worker's cache strategy.

**Flag evaluation in server components.** When we started migrating to Next.js App Router, our existing `useFlags` hooks obviously didn't work in server components. We had to set up the Node SDK separately for server-side evaluation and pass the results down. The two SDKs (client and server) use different flag evaluation contexts, so we had a brief period where server-rendered content showed a different variation than the client-side hydrated version. Not fun.

**Too many flags on one feature.** We once had a feature with four separate flags: one for the UI, one for the API endpoint, one for the analytics tracking, and one for the email notification. Managing the state matrix of 2^4 = 16 possible combinations was a nightmare. Should've been one flag with the backend reading the same flag value. Lesson learned: one feature, one flag, evaluated in as few places as possible.

## When to skip flags entirely

Not every change needs a flag. Bug fixes, copy changes, style tweaks, internal tooling — just ship them. Flags have overhead: the code complexity, the dashboard management, the cleanup. If you're not planning to do a gradual rollout, an A/B test, or a quick kill switch for a risky change, a flag is just ceremony.

I use flags for: new user-facing features on high-traffic pages, migrations that change data flow, and anything touching payments or checkout. For everything else, a regular deploy behind a well-tested PR is simpler and good enough.
