# Introduction

In my journey as a developer, I've always believed that understanding the core architecture of a framework can differentiate you from being just a user to a master craftsman. NestJS, with its robust dependency injection (DI) system, is no exception. This deep dive into NestJS's DI will illuminate the intricacies of the framework, including the internals of the IoC container, module system, provider types, and custom decorators.

## Understanding the IoC Container

The Inversion of Control (IoC) container is at the heart of NestJS's DI system. When you boot up a NestJS application, the IoC container orchestrates the instantiation and resolution of dependencies.

### Bootstrapping and Resolution

- **Singleton Nature**: By default, providers in NestJS are singletons. The IoC container ensures that a single instance is shared across the application.
- **Dependency Graph**: During bootstrap, the container constructs a graph of dependencies, resolving them before the application is fully operational. This ensures that all service dependencies are injected properly.

## Mastering the Module System

Modules in NestJS serve as organizational boundaries and encapsulate related components.

### Global Modules

- **Purpose**: Make services available everywhere without importing the module.
- **Usage**: Use the `@Global()` decorator for services like configuration or logging.

```typescript
import { Module, Global } from '@nestjs/common';

@Global()
@Module({
  providers: [ConfigService],
  exports: [ConfigService],
})
export class ConfigModule {}
```

### Dynamic Modules

Dynamic modules allow the configuration to be passed during import, enabling complex setups.

```typescript
@Module({})
export class AsyncConfigModule {
  static forRoot(options: ConfigOptions): DynamicModule {
    return {
      module: AsyncConfigModule,
      providers: [
        {
          provide: CONFIG_OPTIONS,
          useValue: options,
        },
      ],
    };
  }
}
```

### Lazy-loaded Modules

NestJS doesn't natively support route-based code splitting yet, but you can implement laziness at runtime with dynamic imports and custom strategies.

### Circular Dependency Resolution

Circular dependencies can be thorny.

- **forwardRef**: Allows modules to reference each other in a circular fashion.

```typescript
@Module({
  imports: [forwardRef(() => OtherModule)],
})
export class SomeModule {}
```

- **Re-exports**: Mitigate circular dependencies by re-exporting modules.

## Playing with Provider Types

NestJS offers flexibility with provider types, crucial for designing scalable applications.

### Class Providers

Default provider type using a class as a dependency.

```typescript
@Module({
  providers: [MyService],
})
export class MyModule {}
```

### Value Providers

Provide static value dependencies.

```typescript
providers: [
  {
    provide: 'APP_NAME',
    useValue: 'NestJS Experience',
  },
]
```

### Factory Providers

Construct complex dependencies dynamically.

```typescript
providers: [
  {
    provide: 'ASYNC_CONNECTION',
    useFactory: async (optionsService: OptionsService) => {
      const options = await optionsService.getOptions();
      return new DatabaseConnection(options);
    },
    inject: [OptionsService],
  },
]
```

### Existing Providers

Aliases for existing services.

```typescript
providers: [
  {
    provide: 'MyCustomService',
    useExisting: MyService,
  },
]
```

## Crafting Custom Decorators

Decorators are a powerful feature in NestJS, and creating them is an art form.

### Parameter Decorators

Enhance method parameters dynamically.

```typescript
export const User = createParamDecorator(
  (data: unknown, ctx: ExecutionContext) => {
    const request = ctx.switchToHttp().getRequest();
    return request.user;
  },
);
```

### Combining Decorators and Metadata Reflection

Use metadata to enhance and compose decorators.

```typescript
import 'reflect-metadata';

function Log(target: Object, propertyKey: string) {
  const originalValue = target[propertyKey];

  const logWrapper = (...args: any[]) => {
    console.log(`Called ${propertyKey} with ${args}`);
    return originalValue.apply(this, args);
  };

  Reflect.defineMetadata('log', true, target, propertyKey);
  target[propertyKey] = logWrapper;
}
```

## Circular Dependency Strategies

Beyond `forwardRef`, restructure your modules, decouple services, and consider using interface-based approaches to reduce tight coupling.

## Scoped Injection: Navigate with Care

Understanding injection scopes can save you from performance bottlenecks.

### DEFAULT Scope

The default singleton scope, suitable for shared state.

### REQUEST Scope

Creates a new instance per request, useful for ephemeral data.

### TRANSIENT Scope

Generates a new instance each time it is injected, providing encapsulation.

## Testing with Dependency Injection

Effective testing in NestJS leverages the same principles of DI.

### Overriding Providers

Modify provider implementations for testing purposes.

```typescript
await Test.createTestingModule({
  providers: [
    {
      provide: MyService,
      useClass: MockMyService,
    },
  ],
});
```

### Custom Test Utilities

Construct utilities for commonly overridden services, akin to dependency mocks in unit tests.

```typescript
export const createMyServiceMock = () => ({
  myMethod: jest.fn(),
});
```
