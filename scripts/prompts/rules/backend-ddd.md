## Category: Domain-Driven Design (DDD)

### Code Requirements
- Primary language: TypeScript for tactical patterns (entities, value objects, aggregates, repositories).
- Show DDD building blocks with real domain examples (e-commerce: Order, Payment, Inventory).
- Include aggregate root patterns with invariant enforcement.
- Show domain events, event handlers, and integration events.
- Repository pattern implementations with actual database code (TypeORM/Prisma).
- Include bounded context communication examples.

### Experience Context
- You've applied DDD to decompose complex business domains in e-commerce and fintech.
- You've defined bounded contexts and context maps for systems with 10+ domains.
- You've implemented tactical DDD patterns in TypeScript and Ruby.
- You've experienced the tension between DDD purity and practical delivery — and chosen pragmatism.
- You've used event storming sessions with domain experts to discover bounded contexts.

### Tone Guidance
- DDD is powerful but overengineered when misapplied. Be clear about when it's worth the investment.
- Use concrete business domain examples, not abstract "Entity" and "ValueObject" classes.
- Emphasize ubiquitous language and collaboration with domain experts.
- Acknowledge that most teams adopt DDD partially — and that's okay.
